# Forecasting Privacy-Budget Consumption in Repeated DP-SQL Workloads

Can Öztaş · Engin Demir — Hacettepe University

A differentially private SQL middleware together with a closed-form model that predicts, **before any query runs**, how much privacy budget a workload will spend. Because exact (and provably equivalent) repeats are served from cache by post-processing, the budget is governed by the number of *distinct* queries, `E[u_k] = Σ_i (1 − (1 − p_i)^k)`, which the model forecasts from public repetition structure alone. The forecast is the accounting core of the middleware: it sizes the budget allocator, gates temporal re-releases, and prices equivalence-aware caching.

**Paper:** [`report/paper.pdf`](report/paper.pdf) (LaTeX source: [`report/paper.tex`](report/paper.tex); a self-contained submission bundle is in [`report/journal_submission/`](report/journal_submission/)).

Key results:

- The forecast matches observed budget consumption within 3% in 21 of 24 configurations across a 4,155-trial campaign, and is invariant to data scale and to the per-query budget.
- On 30 workloads derived from real Amazon Redshift traces (Redbench), forecasting the full workload from its first half cuts the budget-prediction error by 45% (0.22 → 0.12) via an unseen-species correction.
- The forecast transfers unchanged to multi-table joins executed live in the middleware; only the per-release noise scale changes.

## Use cases

**Sizing a dashboard's monthly budget before launch.** A team publishes a privacy-protected dashboard: a fixed set of tiles refreshed on a schedule. From the tile distribution alone, the model reports the expected spend `ε_q·E[u_k]` and whether the workload fits the budget `B`; the safe sizing `ε_q = B/m` provably lasts the whole period, since at most `m` distinct queries can ever be charged.

**A go/no-go accuracy check at t = 0.** Before any query runs, the planner answers: "will each released number meet an error target `α` with probability ≥ 1 − β?" A PASS/FAIL verdict is computed from public structure only, so an infeasible privacy/accuracy combination is caught before a single unit of budget is spent.

**Capacity planning from a prefix.** When the workload is unknown, watching only its first half is enough: an unseen-species correction forecasts how many *new* queries are still to come, cutting the budget-prediction error by 45% on real Redshift-derived traces.

**Knowing when caching will not help.** On a drill-down workload where every query is unique, the model predicts zero savings in advance (`S(k) = 0`), so the custodian can plan for naive-composition costs instead of discovering the exhausted budget mid-month.

## Reproducibility

Every number, table, and figure in the paper is regenerated by a script in `experiments/`, with fixed integer seeds.

**1. Install** (Python ≥ 3.10):

```bash
pip install -e ".[dev]"            # core + pytest
pip install -e ".[smt]"            # optional: Z3-backed semantic-equivalence cache
```

**2. Build the datasets** (TPC-H via DuckDB's built-in generator; Adult from UCI):

```bash
mkdir -p data/adult
curl -o data/adult/adult.data https://archive.ics.uci.edu/ml/machine-learning-databases/adult/adult.data
curl -o data/adult/adult.test https://archive.ics.uci.edu/ml/machine-learning-databases/adult/adult.test

python scripts/load_data.py        # → data/dpdb.duckdb  (TPC-H SF=1 + Adult)
python scripts/load_sf10.py        # optional: SF=10 for the scale-invariance sweep
```

**3. Run the test suite** (233 tests, ~10 s):

```bash
python -m pytest -q                # 233 passed
python -m dpdb.model               # model self-check (limit cases)
```

**4. Regenerate the paper's results** (script → result mapping):

```bash
python experiments/model_validation.py --trials 30   # main grid + limits figure
python experiments/redbench_validation.py            # Redbench findings incl. the 45% headline
python experiments/ood_validation.py                 # out-of-distribution check
python experiments/join_validation.py                # twin single-table vs join experiment
python experiments/live_join.py                      # live FK-join end-to-end checks
python experiments/flex_join_forecast.py             # runtime elastic-sensitivity twin
python experiments/equivalence_forecast.py           # equivalence-class forecast
python experiments/markov_occupancy.py               # sticky + general-Markov occupancy
python experiments/markov_modulated.py               # latent-state (HMM) occupancy
python experiments/temporal_bound.py                 # temporal upper bound + tau*(lambda)
python experiments/adaptive_hunting.py               # noise-adaptive "hunting" analyst
python experiments/workload_learning.py              # drift-aware forecasting
python experiments/semantic_normalize.py             # sound equivalence-cache demo
python experiments/smt_semantic.py                   # Z3 SMT equivalence demo (needs .[smt])
python experiments/system_perf.py                    # latency/throughput overhead
python experiments/aggregate_all_results.py          # → results/REPORT.md summary
```

Outputs are written to `results/` as CSVs. The model-validation and allocation experiments are deterministic within a fixed environment (seeds are hard-coded); the large sweeps are distributionally reproducible. An interactive web demo that runs the real pipeline is available via `python demo/app.py`.
